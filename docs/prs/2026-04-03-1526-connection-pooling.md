## Description

Add connection pool manager for upstream TCP/TLS connections (#250)

## Changes

### Connection Pool Infrastructure (initial commit)
- `ConnectionPool` — per-host keyed pool with `ConcurrentQueue`, configurable max per host and idle timeout
- `PooledConnection` — wrapper around `TcpClient`/`Stream` with `ReturnToPool()` and `Dispose()` lifecycle
- Automatic stale connection eviction every 10 seconds via `Timer`
- Health checks using `Socket.Poll` to detect closed connections

### Pool Integration Fix (this commit)
Three bugs fixed that made the pool a no-op:

1. **SslStream Resource Leak**: `DisposeConnection()` only disposed `TcpClient` but never the `Stream`. Every TLS connection going through pool cleanup leaked unmanaged security handles. Fixed: dispose `Stream` before `TcpClient`.

2. **Pool Never Reused Connections**: Both `ForwardHttpRequestAsync` and `HandleTunnelRequestsAsync` sent `Connection: close` and called `Dispose()` in `finally` blocks. `ReturnToPool()` was never called. Fixed: when pool is active, send `Connection: keep-alive`, read responses using proper HTTP framing (Content-Length / chunked), and call `ReturnToPool()` on success.

3. **Double GetStream()**: `CreateConnectionAsync` called `tcpClient.GetStream()` twice — once for the initial stream assignment and again for the `SslStream` wrapper. Fixed: reuse the existing `stream` variable.

## Notes

- When pool is disabled (`ConnectionPoolSizePerHost = 0`), behavior is unchanged (`Connection: close`, read-until-EOF)
- `HandleTunnelRequestsAsync` now loops for multiple requests when pool is active (instead of breaking after one)
- Response framing helper methods (`FindHeaderEndIndex`, `ParseContentLengthFromHeaders`, `EndsWithChunkTerminator`, `IsKeepAliveResponse`) are `internal static` for testability

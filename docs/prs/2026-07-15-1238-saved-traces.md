## Description

Saved Traces view — save/unsave individual traces from Inspection, with notes (#337)

## Scope

- **Backend:** `SavedTrace` + `SavedTraceWebSocketFrame` entities (snapshot copies, independent of sessions), `ISavedTraceRepository`/`SavedTraceRepository`, `SavedTracesController` (`POST/GET/GET{id}/PATCH{id}/DELETE{id}` at `api/saved-traces`), DTOs, DI registration. Table creation via existing `EnsureSchemaCreated` path. Saved traces are exempt from `SessionRetentionService` (it only touches `InspectionSessions`; covered by a regression test).
- **Frontend:**
  - `InspectionRow.SavedTraceId` carries saved state; toggle logic centralized in `InspectionDataService` (`CanSaveTrace`, `ToggleSaveTraceAsync`, `UnsaveTraceAsync`, `OnSavedTracesChanged` event) so the Inspection table, detail panels, and the new page stay in sync.
  - Inspection: `row-saved` accent + star in URL cell, `Save ☆ / Saved ★` toggle in `InspectionDetail` and `WebSocketDetail`, `S` keyboard shortcut (app.js + legend), pending rows blocked from saving.
  - New `/saved-traces` page (sidebar entry): virtualized table (Saved at, Captured at, Method, URL, Status, Size, Duration, Note), search across method/URL/status/note, row click opens the standard detail panels (resend/cURL/WS all work), per-row unsave, notes editable via `TraceNoteEditor` in the detail panels.

## Design decisions

- Saved state lives on the row object (`SavedTraceId`) rather than a separate id map — survives navigation via the singleton service, disappears naturally on Clear/LoadRows, and lets the SavedTraces page reuse the same detail-panel toggle for unsave.
- Notes are edited only from the Saved Traces view (issue's MVP); the detail components take optional `EnableNote`/`Note`/`OnNoteSaved` parameters.
- Option C (hover row action) from the issue was skipped as a fast-follow; A + B implemented.
